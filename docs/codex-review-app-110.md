# codex 리뷰 기록 — APP-110

게이트는 로컬 `codex exec review --base dev` 하나다. PR 원격 Codex 리뷰는 요청·반영하지 않는다.

## 1차 — 목 3종 (초대·알림·멤버 / 회의·분석·연동 / SSE)

7건 전부 유효했고 전부 반영했다.

| 지적 | 판단 | 근거 |
|---|---|---|
| **P1** 테스트 픽스처가 `meetingStartedBy` 누락으로 TS 컴파일 실패 | 고침 | 사실이었다. 그리고 **검증 명령에 구멍이 있었다** — `pnpm build`는 Next가 포함하는 파일만 타입 체크해서 `.test.tsx`가 계약을 어겨도 통과한다. 픽스처를 고치고 `pnpm typecheck`(`tsc --noEmit`)를 스크립트와 검증 명령에 추가했다 |
| **P1** 채팅 REST 핸들러 미등록 | 고침 | `createAgentChat`·`getActiveAgentChat`·`getAgentChatMessages`·`getNoteSharedChatMessages`가 목을 우회하고 있었다. 세션 상태와 히스토리를 `db.ts`에 넣고 네 핸들러를 등록했다 |
| **P1** OAuth authorize·callback이 200 JSON | 고침 | 계약을 확인하니 둘 다 **302 + `Location`**이었다. 200 JSON이면 브라우저에 JSON이 뜨고 왕복이 끝나지 않는다. 302로 바꿨다 |
| **P2** 공유 챗 잠금이 스트림 시작 전에 풀림 | 고침 | 응답 객체를 만들기 전에 풀고 있어서, 첫 스트림이 아직 흐르는 사이 두 번째 요청이 통과했다. 스트림 `finally`에서 풀도록 옮기고 테스트 두 개로 고정했다 — 흐르는 동안 409, 소비 후 200 |
| **P2** 분석 재요청 선행조건 없음 | 고침 | 계약에 `MEETING_NOT_ENDED`·`ANALYSIS_IN_PROGRESS` 409가 있다. 둘 다 넣었다 |
| **P2** 활성 전사 중에도 회의 종료 허용 | 고침 | 계약에 `ACTIVE_TRANSCRIPTION_SESSION` 409가 있다. 종료 전에 세션 상태를 본다 |
| **P2** 초대 시드가 이미 멤버인 워크스페이스를 가리킴 | 고침 | 수락해도 합류를 흉내내지 못하고 멤버만 중복됐다. 아직 멤버가 아닌 워크스페이스(`디자인 팀`)에서 온 초대로 바꾸고, 수락하면 그 워크스페이스가 목록에 나타나게 했다 — 그게 실제 합류다 |

**가장 값어치 있었던 지적은 첫 번째다.** 지적 자체(픽스처 3개)보다 그것이 드러낸 사실이 컸다 — 검증 명령이 테스트 파일의 타입을 보지 않고 있었다. 계약이 넓어질 때마다 픽스처가 조용히 낡는 구조였다.

계약 미러(`openapi3.yml`·`asyncapi.yml`)와 orval 생성물에 대한 지적은 없었다.

## 2차 — Playwright 도입

4건 전부 유효했고 전부 반영했다.

| 지적 | 판단 | 근거 |
|---|---|---|
| **P1** vitest가 Playwright 스펙을 주워 `pnpm test:run`이 깨짐 | 고침 | 사실이었다. 방금 넣은 e2e가 필수 검증 명령을 깨뜨리고 있었다 — `"Playwright Test did not expect test() to be called here"`. `vitest.config.ts`의 `exclude`에 `e2e/**`를 넣었다 |
| **P2** 분석이 없는 노트에 `analyses/latest`가 500 | 고침 | 시드된 모든 노트가 그 상태다. 생성 핸들러가 고정 200이라 예외를 못 잡았다. `commandResult`로 감싸 404로 내보내고, 코드명도 계약의 `ANALYSIS_JOB_NOT_FOUND`로 맞췄다 |
| **P2** 없는 채팅에도 스트림을 열어 줌 | 고침 | 계약은 404 `AGENT_CHAT_NOT_FOUND`다. 스트림을 반쯤 열면 web은 오류로 끝난 스트림을 받는다. 열기 전에 검사하고, 하드코딩된 chatId를 쓰던 테스트를 실제 세션을 만들도록 바꿨다 |
| **P2** 시드 이후 생긴 워크스페이스에 멤버·연동 행이 없음 | 고침 | `createWorkspace`와 초대 수락으로 생긴 워크스페이스가 멤버 목록이 비고 연동 조회가 빈손이었다. `attachWorkspaceState()`로 세 경로(시드·생성·수락)를 한곳에 모았다 |

**첫 번째가 다시 검증 명령 문제였다.** 1차는 `build`가 테스트 파일을 안 본다는 것이었고, 2차는 새 도구가 기존 명령을 깨뜨린 것이다. 검증 명령 자체를 매번 돌려 확인하지 않으면 조용히 무너진다.

`pnpm test:run`을 4회 반복해 안정성을 확인했다(4/4 통과). 중간에 한 번 종료 코드 1이 나왔는데 Playwright dev 서버가 떠 있는 동안 겹쳐 돈 경우였고, 단독 실행에서는 재현되지 않았다.

## 3차 — 브랜치 전체

4건 전부 유효했고 전부 반영했다.

| 지적 | 판단 | 근거 |
|---|---|---|
| **P2** 승인을 실제로 기다리지 않음 | 고침 | **이번 리뷰에서 가장 값어치 있는 지적이다.** 목이 `tool_approval_request` 40ms 뒤에 스스로 승인해 버려서, 승인 API를 부를 틈도 거절을 밟을 방법도 없었다. 승인 UI를 데모하려고 만든 목인데 정작 그 대기가 없었다. 스트림을 `before` / `after(decision)`로 쪼개고 핸들러가 승인 API 호출(또는 타임아웃)까지 **실제로 멈추게** 했다. 거절이면 도구를 실행하지 않고 정상 종료한다 |
| **P2** 첫 녹음 시작 시 `meetingStartedBy`가 안 채워짐 | 고침 | 계약이 "녹음을 최초로 시작한 유저"로 정의하는데 계속 null이었다. 시작자만 회의를 끝낼 수 있으므로 그 판정의 근거가 없던 셈이다 |
| **P2** 읽기 핸들러의 미스가 MSW 500 | 고침 | 멤버·초대·연동·알림읽음·채팅 조회가 예외를 그대로 던져 500이 됐다. 계약의 404 JSON으로 내보내도록 `commandResult`로 감쌌다 |
| **P2** `markNotificationRead`가 알림 전체를 반환 | 고침 | 계약은 `{ notificationId }`만이다. 전체를 흘리면 UI가 실제 응답에 없는 필드에 기대게 된다 |

## 4차 — 재검토 (3차 반영 뒤)

6건 더 나왔고 전부 유효했다. 반영 후 검증 통과.

| 지적 | 판단 | 근거 |
|---|---|---|
| **P2** `/mock-oauth` 라우트가 없어 302가 Next 404로 떨어짐 | 고침 | 목의 OAuth 왕복이 아예 닫히지 않고 있었다. 화면을 만들었고, 만들면서 **MSW가 최상위 내비게이션을 가로채지 못한다**는 사실이 드러났다 — 콜백을 `<a href>`로 걸면 브라우저가 콜백 URL에 그대로 멈춘다. 목 화면이 콜백을 fetch로 부르고 이동은 라우터가 하도록 바꿨다. Playwright로 왕복이 닫히는 것을 확인했다 |
| **P2** `pauseMeeting`에 활성 전사 검사 없음 | 고침 | 종료와 같은 규칙인데 중지에만 빠져 있었다. 녹음 중인 채로 PAUSED가 되는 상태를 목이 허용했다 |
| **P2** 공유 챗 `pendingApproval`이 항상 null | 고침 | **관전자 폴링의 유일한 창구다**(계약). 승인 대기 중에도 null이라 관전자 화면을 만들 근거가 없었다. 스트림이 기다리는 동안 채우고 끝나면 지운다 |
| **P2** 승인 API가 200 JSON | 고침 | 계약은 bodyless 204다. 생성 클라이언트가 서버와 다르게 굴러간다 |
| **P2** `NOTIFICATION_NOT_FOUND`·`AGENT_CHAT_NOT_FOUND`가 409로 나감 | 고침 | 404 집합에서 빠져 있었다. 클라이언트 오류 처리가 충돌 경로로 샌다 |
| **P2** TOOL 히스토리에서 승인과 실행이 한 줄로 뭉침 | 고침 | 계약은 승인 기록(`decision`, `status` null)과 실행 기록(`status`, `decision` null)을 나눈다. 게다가 **거절은 아무 기록도 남기지 않고 있었다** — 새로고침하면 거절한 사실이 사라졌다 |

## 6차 — 재검토 (5차 반영 뒤)

6건 나왔고 전부 유효했다. 여기서 게이트를 닫는다.

| 지적 | 판단 | 근거 |
|---|---|---|
| **P1** 승인 상한 1.5초라 사람이 누를 수 없음 | 고침 | **내가 만든 충돌이다.** 3차에서 만료 경로를 테스트하려고 상한을 낮췄는데, 그 값이 데모에도 적용돼 카드를 읽기도 전에 자동 거절되고 뒤늦은 클릭은 404가 됐다. 기본값을 120초로 올리고 테스트만 `setApprovalTimeoutForTests()`로 짧게 덮는다 |
| **P2** 공유 챗 ACTIVE 판정에 시작자가 빠짐 | 고침 | 계약의 ACTIVE는 `IN_PROGRESS + 시작자 존재`다. 새 노트는 생성 시부터 IN_PROGRESS라 아직 회의가 열리지도 않았는데 공유 챗이 열렸다 |
| **P2** 같은 채팅의 두 번째 응답이 첫 응답의 id를 재사용 | 고침 | `messageId`·`approvalId`·`toolCallId`가 `chatId`에서만 나왔다. 멀티턴에서 id로 메시지를 맞추는 클라이언트가 앞 턴을 덮어쓴다. 턴 번호를 seed에 넣었다 |
| **P2** 연동 해제가 200 JSON | 고침 | 계약은 bodyless 204다 |
| **P2** enum 밖 `decision`을 APPROVED로 처리 | 고침 | 계약은 400이다. **오타가 쓰기 도구를 실행하는 기본값**이었다 |
| **P2** 빈 공유 챗 메시지가 잠금을 잡고 스트림을 염 | 고침 | 계약은 400이다. 잠금을 잡기 전에 막아야 실패한 요청이 잠금을 남기지 않는다 |

이 라운드에서 테스트도 함께 고쳤다. 승인 id를 바깥에서 추측하던 테스트를 **스트림에서 읽거나
관전자 폴링(`lock.pendingApproval`)에서 얻도록** 바꿨다 — 턴마다 id가 달라진 뒤로 추측이
성립하지 않고, 실제 클라이언트가 하는 방식이기도 하다.

## 여섯 게이트를 통과하며 배운 것

여섯 번 모두 **목이 계약보다 관대했다**는 지적이 나왔다. 목이 통과시키는 것을 실제 서버는 거절한다면, 그 목으로 통과한 화면 테스트는 거짓 안심이다.

가장 값어치 있던 셋은 전부 **목이 흉내만 내고 있던 것**이었다 — 승인을 기다리지 않았고,
관전자에게 보일 대기 상태가 없었고, 거절 기록이 남지 않았다. 셋 다 화면을 만들 근거가
없다는 뜻이고, 목으로 통과한 테스트는 거짓 안심이 된다.

한 번은 **내가 만든 충돌**이 잡혔다 — 테스트를 통과시키려 낮춘 승인 상한이 데모를
쓸 수 없게 만들었다. 테스트 편의를 위한 값이 제품 동작이 되는 자리는 리뷰가 아니면
잘 안 보인다.

그리고 1·2차의 첫 지적이 둘 다 **검증 명령 자체의 문제**였다 — `build`가 테스트 파일을 안 보고, 새로 넣은 도구가 기존 명령을 깨뜨렸다. 명령을 매번 끝까지 돌려 확인하지 않으면 조용히 무너진다.
