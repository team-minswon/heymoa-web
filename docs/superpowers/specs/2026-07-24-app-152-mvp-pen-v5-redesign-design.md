# APP-152 mvp.pen v5 — 전체 화면 재디자인 설계

**목표:** heymoa-web의 전 화면을 mvp.pen v5 행에 다시 그린다. v5는 시안이고, 구현이 끝나면 실제 앱이 정답이 된다.

**입력:** MSW 실측 화면(2026-07-24, 1440×900), v4 행(y17000~36900)과 그 SPEC 노트 `LNplj`·`kuPpg`·`ROX9B`, `DESIGN.md`, `openapi3.yml`, 실제 라우트·컴포넌트 트리.

**범위 밖:** 코드 구현 전부. 토큰 반영은 APP-153, 화면 구현은 APP-154~157. v4 행 삭제(참조용 보존). 계약 변경.

## 자문자답으로 잡은 것

### "디자인이 섞였다"는 스타일 문제가 아니다 — 캔버스와 코드가 다른 세대다

MSW로 앱을 띄워 v4 결정과 대조한 결과, 갈라짐의 정체가 취향 불일치가 아니라 **v4에서 확정한 값이 코드에 반영되지 않은 것**으로 확인됐다.

| 항목 | v4 결정 | 실제 코드 (2026-07-24 실측) |
|---|---|---|
| 목록 행 | 높이 52 · 한 줄 · r8 · 배경 없음 · 행 사이 hairline | 높이 ~90 · 흰 카드 · pill radius · 그림자 |
| 챗봇 표면 | 트레이 안 부양 카드 416×788 r16 + 2연타 그림자 | 노트 우측에 붙은 붙박이 컬럼, 카드 아님 |
| full 탭 | 좌측정렬 · 라벨폭 밑줄 | 전체폭 3등분 균등 (side 규칙이 full에 적용됨) |
| 개인 챗봇 그림자 | e2 2연타 | `shadow-[0_4px_16px_rgba(0,0,0,0.04)]` — `kuPpg`가 "흰 마케팅 페이지용이라 제품 캔버스에서 안 보인다"고 **명시적으로 배제한 값** |
| 설정 다이얼로그 | 형태 스케일 5단계(16/10/8/6/full) | `rounded-[28px]` — 스케일에 없는 값 |

**그래서 v5의 일은 새 취향을 정하는 것이 아니라, 흩어진 결정을 한 곳에서 다시 못박고 구현이 따라올 수 있는 형태로 남기는 것이다.** 이 표의 각 항목은 v5 SPEC 노트가 개별적으로 재확정한다.

### 밀도 합격선을 어느 화면도 통과하지 못한다

v4가 세운 기준은 "주 콘텐츠 8개 이상, 채팅 2왕복 이상, 200px 넘는 빈 세로 띠 없음"이다. 실측 결과:

- 노트 허브: 본문 아래 ~600px 공백 (노트 2개)
- 노트 상세 full: 전사 1행 아래 ~600px 공백
- 공유 챗봇: ~750px 공백

**빈 공간의 원인은 레이아웃이 아니라 목 데이터의 빈약함이기도 하다.** 그래서 v5 프레임은 합격선을 채우는 데이터로 그리고, 목 보강은 APP-156이 받는다. 프레임을 빈 상태로 그리면 밀도 판단이 불가능하다.

### 마케팅 조판이 제품 화면에 들어와 있다

`MEETING NOTES / 모든 회의`, `CONVERSATION / 대화 기록` — 대문자 키커 + 세리프 제목의 잡지 표지 조판이다. 특히 후자는 탭이 이미 "실시간 전사"라고 말한 바로 아래에서 같은 말을 반복한다.

**대문자 키커는 제품 면에서 전부 뺀다.** 탭·브레드크럼·사이드바가 이미 위치를 말하고 있어 키커는 중복이고, 세로 공간을 먹으면서 밀도를 떨어뜨린다. **세리프 300 제목은 남긴다** — 화면 제목과 노트 제목에서 heymoa의 정체성을 지탱하는 유일한 요소이고, 이것까지 빼면 shadcn 기본값 티가 난다.

랜딩·약관은 마케팅 면이므로 키커를 유지한다. **이 분리가 `DESIGN.md`에 없어서 지금까지 화면마다 다시 결정됐다** — APP-153이 문서화한다.

### full 모드가 워크스페이스를 덮어 내비게이션을 잃는다

실측상 노트 상세 full은 좌측 사이드바까지 덮는 전체폭 오버레이다(접근성 트리에는 워크스페이스 노트 링크가 남아 있으나 화면에 없다). 노트를 하나 열면 다른 노트로 갈 길이 상단 브레드크럼뿐이다.

**full에서도 사이드바 255를 유지한다.** `kuPpg`가 적은 원래 구조가 이것이고("워크스페이스 사이드바 255 + 메인 1185"), 사이드바를 남기면 상단 크롬 2단 문제도 같이 풀린다 — 노트 액션을 상단바로 올릴 자리가 생긴다.

### 상단 크롬이 2단이라 알림이 동떨어져 보인다

상단바는 좌측 브레드크럼과 우측 끝 벨뿐이고 가운데 ~1100px가 비었는데, 노트 액션(중지·회의 종료·패널 토글)은 그 아래 별도 행에 있다. 크롬이 두 겹이다.

**노트 액션을 상단바로 올려 1단으로 합친다. 벨은 우측 끝 그대로 둔다.** 알림 위치가 틀렸던 게 아니라 주변이 비어서 동떨어져 보였던 것이다. 통합으로 세로 64px를 회수한다.

### 사이드바 상단에 스위처가 두 개 쌓여 있다

유저 카드(64) + 워크스페이스 스위처(56) = 120px. 둘 다 "무언가를 바꾸는" 컨트롤인데 위아래로 붙어 있어 위계가 읽히지 않는다.

**상단은 워크스페이스 1줄(56)만 두고, 유저 아바타·설정은 사이드바 최하단으로 내린다.** Linear·Slack의 배치다. 목록에 64px가 회수된다.

### 설정 계층 push 스펙(`LNplj`)은 v5가 계승하지 않는다

`LNplj`는 Cloudflare 대시보드를 100ms 간격으로 계측해 만든 260ms 사이드바 push/pop 명세다. 그런데 **설정은 라우트가 아니라 전체화면급 Dialog다**(`max-w-5xl`, `h-[min(760px,...)]`). 사이드바가 밀려 들어올 구조가 앱에 없다.

**다이얼로그를 유지하고 push 스펙을 폐기한다.** 없는 구조를 정당화하려고 설정을 라우트로 바꾸는 것은 순서가 거꾸로다. 대신 다이얼로그 자체를 형태 스케일과 e3 고도에 맞춘다(`rounded-[28px]` → 스케일 값). v5 SPEC 노트에 폐기 사유를 남겨 다음 사람이 "왜 이 스펙이 안 쓰이지"를 다시 조사하지 않게 한다.

**`ROX9B`(레코더)와 `kuPpg`(고도)는 계승한다.** 레코더 3표면과 고도 3단은 실제 코드에 대응물이 있고 유효하다.

### 중지 버튼이 두 개다

노트 상세 헤더의 "중지"와 하단 레코더 독의 붉은 원이 같은 동작이다. **레코더 독 하나로 줄인다** — 독은 `ROX9B`가 상태별 폭 전환까지 정의한 표면이고, 헤더 버튼은 그 기능의 중복 표기다.

### 알림 드롭다운이 반투명하다

실측에서 드롭다운 뒤 챗봇 텍스트가 그대로 비쳐 글자가 겹쳤다. 취향이 아니라 결함이다. v5는 불투명 배경 + e3 고도로 그린다.

## 캔버스 구조

v5 행 `y=40000~`. v4는 보존한다. 프레임 피치 x1840, 행 라벨 rowY-120 — v4 관례 계승.

| 행 | 내용 |
|---|---|
| y40000 | 셸 기준 — 사이드바 확정안, 상단바 1단, 워크스페이스 노트 허브 |
| y42000 | 노트 상세 full — 전사 / 요약 / 노트 정보 |
| y44000 | 노트 상세 side 시트 |
| y46000 | 챗봇 2종 — 공유 트레이 / 개인 FAB·카드 / 도구 승인 |
| y48000 | 알림 드롭다운 · 설정 다이얼로그(계정·워크스페이스·멤버·연동) |
| y50000 | 상태 화면 — 빈 상태 · 스켈레톤 · 에러/재시도 · 회의 종료 |
| y52000 | SPEC 노트 + 키프레임 |

y52000은 v4 모션 행(`DW8uc` 라벨 + note + 키프레임 프레임)의 형식을 그대로 계승한다. 담을 노트:

- **FORM SPEC** — 형태 스케일 · 타이포 스케일 · 목록 행 · 밀도 합격선. 위 drift 표의 각 항목을 재확정
- **ELEVATION SPEC** — `kuPpg` 계승·개정판
- **MOTION SPEC** — `ROX9B` 계승(레코더), `LNplj` 폐기 사유 명시
- **CHROME SPEC** — 사이드바 구성 · 상단바 1단 · 알림 위치

## 실행 — 에이전트 6개 병렬

y40000~50000 여섯 행을 1행 1에이전트로 병렬 작업한다(APP-145 전례). **y52000 SPEC 노트는 여섯 결과가 나온 뒤 통합 작성한다** — 수치를 여섯 곳에서 각자 적으면 지금과 같은 갈라짐이 재발한다.

각 에이전트에 공통으로 주는 것: 확정된 방향 6개, drift 표, 밀도 합격선, 형태·타이포 스케일, 계약에 없는 데이터 금지.

**형태·타이포 스케일은 v4 값을 그대로 상속한다** — 형태 panel 16 / block 10 / control 8 / chip 6 / circle full, pill은 주 CTA와 레코더 독만. 타이포 화면 제목 34 / 노트 제목 26 / 섹션 20 / 패널 제목 18 / 전사 본문 15 / 목록 행 제목 15. 위 drift 표가 바꾸는 항목만 예외다. 이 값들은 이미 합의된 것이므로 v5에서 다시 정하지 않는다 — 다시 정하면 그 자체가 새 갈라짐이다.

## 완료 조건

- [ ] v5 행에 앱 전 화면 프레임이 있고 이름이 `v5/...` 규칙을 따른다
- [ ] drift 표의 7개 항목이 v5에서 전부 해소됐고 SPEC 노트가 각각을 수치로 재확정한다
- [ ] 모든 프레임이 밀도 합격선을 통과한다 — 200px 넘는 빈 세로 띠 없음
- [ ] 제품 면에 대문자 키커가 없고, 화면 제목·노트 제목의 세리프 300은 남아 있다
- [ ] full 모드 프레임에 사이드바 255가 있고 상단 크롬이 1단이다
- [ ] `LNplj` 폐기 사유가 SPEC 노트에 남아 있다
- [ ] 모든 프레임의 데이터가 `openapi3.yml`로 채울 수 있다
- [ ] `docs/design-decisions.md` v5 프레임 ID 표 갱신
