# APP-144 API 맥락 맵 설계

**목표:** `openapi3.yml`이 만드는 화면·상태·실패를 한 문서로 모아, v4 디자인(APP-145)의 여섯 에이전트와 화면 구현(APP-111~117)이 같은 것을 읽게 한다.

**범위 밖:** v4 프레임을 그리는 일(APP-145), 화면 구현(APP-111~117).

## 배경

APP-110이 34경로·44 operation을 web이 호출할 수 있게 만들었다. 이제 **그 호출이 무엇을 만드는지**가 어디에도 정리돼 있지 않다. 여섯 디자인 에이전트가 각자 계약을 읽으면 각자 다른 결론에 이른다.

## 결정

### 행 단위는 operation이다

34경로에 44 operation이 있고 **훅은 operation 단위**로 생성된다. 경로 단위로 묶으면 `GET`과 `POST`가 한 줄에 뭉쳐 훅과 상태가 대응되지 않는다.

### `v4 프레임 ID`는 전부 `없음`으로 시작한다

지금 v4 프레임은 APP-145가 손대기 전 상태다. 여기서 기존 25프레임을 미리 매핑해 두면 APP-145가 재구성할 때 다시 써야 한다. **`없음`의 집합이 곧 APP-145의 작업 목록**이고, APP-145가 그리면서 채운다.

이 결정의 대가는 지금 이 문서만으로는 "무엇이 이미 있는가"를 알 수 없다는 것이다. 대신 APP-145가 자기 행의 프레임을 조회하면서 채우므로, 한 번만 조회하면 된다.

### 실패를 화면 관점으로 적는다

계약의 상태 코드를 그대로 옮기면 디자이너에게 쓸모가 없다. **그 실패가 화면에서 무엇으로 보여야 하는가**를 함께 적는다. 특히 계약이 명시한 "종료 이벤트 없이 끊기는 SSE"처럼 코드로 표현되지 않는 실패를 빠뜨리지 않는다.

### shadcn 매핑을 같은 문서에 둔다

여섯이 각자 프리미티브를 고르면 구현이 여섯 갈래로 갈라진다. 별도 문서로 빼면 안 읽는다.

## 성공 기준

- 44 operation이 전부 표에 있다 (34경로)
- 각 행에 훅 이름이 있고, **생성 파일에서 확인한 실제 이름**이다
- 계약이 만드는 실패가 화면 관점으로 정리돼 있다 — SSE의 세 번째 종료 경로 포함
- shadcn 매핑표가 있다
- 생성 훅을 쓸 수 없는 세 operation(SSE 둘 + `startWorkspaceIntegration`)이 그 사실과 함께 표시돼 있다
