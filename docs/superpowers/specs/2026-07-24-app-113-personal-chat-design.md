# APP-113 개인 챗봇 UI 설계

**목표:** 개인 챗봇을 글로벌 플로팅 패널로 붙이고, **SSE 클라이언트를 여기서 만든다** — APP-112(공유 챗봇)와 APP-114(승인 UX)가 그대로 재사용한다.

**범위 밖:** 공유 챗봇(APP-112), 승인 UX의 상태들(APP-114 — 204 접수 잠금·만료 무효화·403/404/409 분기·관전자 pending row).

**입력:** `docs/api-surface.md` 개인 챗봇 절(4 operation + SSE), `docs/design-decisions.md` 개인 챗봇 행(프레임 `LeuWE` `LCXcj` `xyR27` `evks6` `L0Wx2` `Z3EDew` `tmNgz` `pn9GH`), `docs/contracts/asyncapi-web-server.yml`의 SSE 이벤트 8종.

## 자문자답으로 잡은 것

사용자에게 묻지 않고 결정했다. 그 과정에서 **설계 문서가 스스로 미뤄 둔 결정 둘**과 **계약이 만드는 함정 하나**가 드러났다.

### 패널을 워크스페이스는 절대 좌표, 노트는 flex 트레이로 둘 수 없다

`design-decisions.md`가 직접 적었다 — "워크스페이스 화면의 패널은 절대 좌표(x=984), 노트 화면은 flex 트레이(464 + padding 8)로 구조가 다르다. 결과 폭은 448로 같아 시각적으로는 일치하지만, 구현 시 한쪽으로 통일하는 게 낫다."

**통일한다: `fixed` 오버레이 하나.** 폭 448, 우측 거터 8. 근거는 셋이다.

1. 개인 챗봇은 **글로벌**이다. 워크스페이스 화면·노트 full 두 라우트에 각각 트레이를 심으면 같은 패널이 두 벌이 되고, 라우트를 옮길 때 열린 상태와 스트림이 끊긴다.
2. 두 화면의 결과 폭이 448로 같으므로 `fixed` 하나로 두 프레임을 모두 만족한다.
3. 본문이 밀리는 것(워크스페이스 본문 832 → 721)은 셸의 우측 padding 한 줄로 낸다. 레이아웃 구조를 갈라 놓는 것보다 diff가 작다.

스트림이 라우트 전환을 견뎌야 하므로 패널은 **워크스페이스 라우트 레이아웃**(`WorkspaceAppShell`)에 산다. 노트 페이지는 그 안에 있다.

### side 모드는 "숨김"이지 "언마운트"가 아니다

노트 side 모드(Sheet)에서 패널을 숨긴다는 건 계약이 아니라 화면 결정이다. 스트림이 흐르는 중에 유저가 노트를 side로 열면?

**흐르는 스트림은 끊지 않고, 보이는 것만 감춘다.** 스트림을 끊으면 유저가 시작하지 않은 중단이 되고, 계약상 부분 응답은 저장되지 않으므로 답변이 통째로 사라진다. side를 닫으면 패널이 그대로 다시 보인다.

### 40초 유휴 타이머는 승인 대기를 오탐한다

세 번째 종료 경로(**종료 이벤트 없이 끊김**)를 프레임 `pn9GH`가 "40초째 멈춰 있습니다"로 표현한다. 그대로 유휴 타이머를 걸면 **승인 대기에서 오탐한다** — 계약의 승인 대기 상한은 300초이고, 그동안 스트림에는 keepalive 주석만 흐른다.

**유휴 타이머는 `tool_approval_request`와 `tool_approval_resolved` 사이에서 멈춘다.** 승인을 기다리는 건 멈춘 게 아니라 기다리는 것이고, 화면도 이미 "승인 대기"라고 말하고 있다.

정지 상태(`stalled`)로 들어가는 길은 둘인데 화면은 하나다.

| 트리거 | 언제 |
|---|---|
| 리더가 종료 이벤트 없이 닫힘 | 목·서버가 스트림을 그냥 끊었을 때 |
| 유휴 40초 (승인 대기 중엔 정지) | 열려 있는데 아무것도 안 올 때 |

둘 다 부분 토큰을 **회색으로 남기고** 재시도를 제안한다. `pn9GH` 한 프레임이 둘을 덮는다.

### 세 종료 경로가 부분 토큰을 서로 다르게 다룬다

계약을 화면으로 옮기면 이렇게 갈린다. 이게 리듀서 설계의 전부다.

| 종료 | 부분 토큰 | 최종 텍스트 | 화면 |
|---|---|---|---|
| `message_end` | 버린다 | **`message_end.content`** | 정상 말풍선 (`L0Wx2`) |
| `error` | **버린다** | 없음 | 말풍선 없이 `Alert`(destructive) + 다시 보내기 (`tmNgz`) |
| 종료 이벤트 없음 | **남긴다(회색)** | 없음 | `Alert`(중립) + 다시 보내기 (`pn9GH`) |

`message_end`에서 토큰 합을 버리고 `content`를 쓰는 게 핵심이다. 계약이 "둘은 같다"고 말하지만, 같다고 **믿고** 토큰 합을 남기면 새로고침 후 글이 바뀌는 버그가 조용히 산다. 히스토리에 남는 것은 `content`뿐이다.

### `tool_call_result(status=error)`는 종료가 아니다

프레임 `Z3EDew`가 이걸 위해 있다. 실패 Badge를 남기고 **토큰이 계속 이어진다**. 리듀서는 `error` 이벤트와 `tool_call_result.status === "error"`를 절대 같은 갈래로 두지 않는다.

### `GET /v1/agent-chats/active`의 `null`은 오류가 아니다

`data: null`이 200으로 온다. 오류 화면(`error boundary`)이 아니라 빈 상태(`evks6`)다. 첫 전송이 `POST /v1/agent-chats`를 불러 세션을 만든다.

즉 **전송은 두 갈래**다.

```
활성 세션 있음 → POST /v1/agent-chats/{chatId}/messages (스트림)
활성 세션 없음 → POST /v1/agent-chats (세션 생성) → 위와 같음
```

### 승인 버튼은 APP-113이 갖는다

승인 UX는 APP-114지만, 목은 "이슈"가 든 메시지에서 **실제로 멈춰** 승인을 기다린다(`sse-handler.ts`). APP-113만 머지된 상태에서 버튼이 없으면 데모가 120초 멈춘 뒤 자동 거절된다 — 머지 가능한 브랜치가 아니다.

**경계를 이렇게 긋는다.**

- APP-113: `pendingApproval` 상태 + 승인/거절 버튼 + `useResolveToolApproval` 호출. 스트림이 `tool_approval_resolved`를 줄 때까지 컴포저는 잠긴다.
- APP-114: 204가 확정이 아님을 드러내는 잠금 표현(`pysh5`), 만료 무효화(`WKrCG`), 403/404/409 분기(`d9IWR`), 관전자 pending row(`jobCE`), 승인/실행 기록 분리 심화(`ImOW0`), 300초 상한 문구.

### TOOL 메시지는 두 종류이고, 지금 갈라야 한다

`L0Wx2`가 APP-113의 성공 기준(새로고침 후 복원)에 속하므로 히스토리 렌더는 여기서 만든다.

- `toolEvent.decision !== null` → **승인 기록**. 좌측 2px 규칙선.
- `toolEvent.status !== null` → **실행 기록**. `Card` 축약 + 상태 Badge + `url` 있으면 외부 링크.

계약상 둘은 배타다(하나가 있으면 다른 하나는 `null`). 판별을 `decision`으로 하고 나머지를 실행 기록으로 떨어뜨리면, 둘 다 `null`인 미래 형태가 실행 기록으로 새어 든다. **둘 다 명시적으로 검사하고 어느 쪽도 아니면 렌더하지 않는다.**

### 스코프는 패널이 정하지 않는다

`chevron-down`을 스코프 전환으로 읽는 문제를 `design-decisions.md`가 보고만 하고 미뤘다. **드롭다운을 만들지 않는다.** 스코프는 "어디서 열었는가"로 정해진다 — 워크스페이스 화면이면 `workspace`, 노트 full이면 `note`. 패널은 현재 스코프를 **읽기 전용 라벨**로 보인다. 어포던스가 없으면 오해도 없다.

## 구조

```
lib/chat/stream-protocol.ts     SSE 이벤트 → 상태. 순수 함수. 브라우저 없이 테스트된다
lib/chat/use-chat-stream.ts     postEventStream을 리듀서에 물리는 훅. abort·유휴 타이머
components/chat/chat-thread.tsx 히스토리 + 진행 중 턴 + 도구 기록 + 승인 행 렌더
components/chat/personal-chat.tsx  Provider(스코프·열림) + 플로팅 버튼 + 패널 + 컴포저
```

앞의 셋은 공유 챗봇(APP-112)이 그대로 쓴다 — 이벤트 계약이 같고 URL만 다르다. 그래서 훅은 URL과 body를 인자로 받고, 스레드는 메시지 배열을 받는다. 개인/공유를 아는 것은 `personal-chat.tsx` 하나뿐이다.

### 상태 기계

```
idle ──send──▶ streaming ──message_end──▶ idle (말풍선 확정)
                  │
                  ├─ error ────────────▶ failed  (부분 버림)
                  ├─ 리더 종료(무종료) ─▶ stalled (부분 회색 유지)
                  ├─ 유휴 40초 ────────▶ stalled
                  ├─ tool_approval_request ─▶ awaiting_approval (유휴 타이머 정지)
                  └─ 유저 중지 ─────────▶ aborted (부분 회색 유지)
```

`aborted`는 유저가 만든 중단이므로 `Alert` 없이 재전송 버튼만 남긴다.

## 오류 표시

`AGENTS.md`의 경계표를 따른다.

| 무엇 | 어떻게 |
|---|---|
| 스트림 `error` · 무종료 · 중단 | 스레드 안 인라인 `Alert` + 재전송 (주 데이터가 없음) |
| 세션 생성 실패 · 승인 실패 | sonner 토스트 (전역 `MutationCache`) |
| 히스토리 404 | 빈 상태로 떨어뜨리고 새 대화 유도 (`evks6`와 같은 화면) |
| `active` 조회 로딩 | 스레드 `Skeleton` |

스트림 실패는 `useChatStream`이 만드는 상태라 mutation 토스트 경로를 지나지 않는다. 승인 mutation만 `meta.suppressErrorToast` 없이 기본 토스트를 쓴다.

## 성공 기준

- 스트리밍 토큰이 실시간으로 붙고, `message_end` 뒤 텍스트가 `content`와 일치한다
- 세 종료 경로가 각각 다른 화면을 만든다 (`tmNgz` / `pn9GH` / 정상)
- `tool_call_result(status=error)` 뒤로 토큰이 계속 붙는다
- 새로고침 후 `active` → `messages`로 대화가 복원되고, TOOL 두 종류가 갈라져 보인다
- `active`가 `null`이면 빈 상태, 첫 전송이 세션을 만든다
- "새로운 대화 시작"이 새 세션을 만들고 스레드를 비운다
- 노트 side 모드에서 패널·버튼이 보이지 않고, 흐르던 스트림은 끊기지 않는다
- 패널이 현재 스코프를 정확히 표시한다 (워크스페이스 / 노트)
- vitest: 리듀서 단위 + 훅(목 스트림) + 패널 통합
- Playwright 스모크: 패널을 열어 한 턴을 끝까지 흘린다
