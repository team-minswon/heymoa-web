# 계약 반영과 v4 디자인 기반 설계

> **이 문서는 로드맵이다.** 2026-07-23부터 이슈마다 자기 spec·plan을 갖는다 —
> 이 문서 하나가 이슈 셋을 덮고 있어서, 이슈를 열어도 자기 설계가 없고 큰 문서의
> 몇 단계를 가리킬 뿐이었다. 여기 남는 것은 **이슈를 가로지르는 결정과 순서**다.
>
> - APP-110 계약 반영 → `2026-07-23-app-110-contract-sync-design.md` / `plans/2026-07-23-app-110-contract-sync.md`
> - APP-144 API 맥락 맵 → 착수 시 brainstorming으로 작성
> - APP-145 v4 계약 정합성 감사 → 착수 시 brainstorming으로 작성

**목표:** heymoa-server가 구현을 끝낸 API 전부를 web이 호출할 수 있게 만들고, 그 호출 가능한 훅을 근거로 mvp.pen v4를 화면 구현이 바로 가능한 상태까지 채운다.

**범위 밖:** 화면 구현(APP-111~117). 이 스펙은 그 일곱 개가 딛고 설 바닥만 만든다.

## 배경

2026-07-23 기준 heymoa-server `dev`가 기획 v2 체인(APP-102)과 멤버 초대(APP-63)를 모두 병합했다. `docs/contracts/openapi3-server.yml` 미러는 37경로인데 `heymoa-web/openapi3.yml`은 13경로다. **24경로가 web에 없다.**

같은 날 `docs/contracts`가 경계별 4파일로 재편됐다.

| 파일 | 경계 |
|---|---|
| `openapi3-server.yml` | web → server public REST (생성물 미러) |
| `asyncapi-web-server.yml` | web ↔ server 비동기 (전사 STOMP + agent 채팅 SSE) |
| `asyncapi-server-ai.yml` | server ↔ ai 채팅 SSE (계획) |
| `openapi3-ai.yml` | server → ai 내부 REST (계획) |

이 재편으로 **APP-110의 전제가 무너졌다.** APP-110은 "아직 서버가 구현 안 한 경로를 확정된 계약 초안으로 오버레이"하는 이슈인데, 오버레이 소스인 `heymoa-server.v2-draft.openapi.yml`이 삭제됐고(서버가 다 구현해서 미러에 흡수됨) 참조 브랜치 `feature/app-63`도 `dev`에 병합됐다. 첨부 3개 중 2개가 없는 파일을 가리키고, 성공 기준 3번("v2-draft와 diff로 정합성 검증하는 절차 문서화")은 대상이 사라졌다. 오버레이 없이 그냥 복사하면 된다.

## 단계

### 0. Linear 이슈 재배치

낡은 이슈 위에 계획을 세우면 계획도 같이 낡는다. 그래서 이 단계가 먼저다.

**지금 이 스펙의 단계와 Linear 이슈는 맞지 않는다.**

| 단계 | 담을 이슈 | 상태 |
|---|---|---|
| 1~3 계약·orval·목 | APP-110 | 본문이 낡음 — 재작성 |
| 4 API 맥락 맵 | 없음 | 신규 필요 |
| 5 v4 계약 정합성 | 없음 | 신규 필요 |

APP-109(`mvp.pen 신규 화면 디자인`)가 2026-07-22에 완료되며 v4 25프레임을 만들었다. 5단계는 그 결과물을 **계약 기준으로 다시 감사하고 구멍을 메우는** 별개 작업이라 완료된 이슈에 얹을 수 없다.

할 일:

- **APP-110 재작성** — 제목에서 "오버레이"를 빼고 본문을 1~3단계로 교체한다. 죽은 첨부 2개(v2-draft, 구 미러 경로)를 제거하고 새 계약 파일 링크로 갈아끼운다.
- **신규 이슈 2개** — 4단계(API 맥락 맵 + shadcn 매핑표), 5단계(v4 계약 정합성 감사, 에이전트 6개 병렬). 둘 다 APP-102 하위, `heymoa-web` 라벨.
- **APP-109는 Done으로 둔다.** 되돌리면 "v4를 그렸다"는 완료된 사실이 지워진다. 신규 5단계 이슈에서 후속으로 참조만 한다.
- **APP-111~117 참조 감사 + 의존 재설정** — 각 이슈가 인용하는 계약 파일명·경로·서버 이슈 상태를 현재와 대조해 낡은 참조를 고친다. 요구사항 자체는 건드리지 않는다. 그리고 일곱 개 전부 신규 5단계 이슈에 블록됨을 명시하고, 우선순위를 새 이슈들 뒤로 조정한다. 요구가 실제로 바뀐 것이 발견되면 코멘트로 남기고 별도 판단 대상으로 올린다 — 이 단계에서 임의로 고치지 않는다.

산출물: 재배치된 Linear 이슈(수정 8개 + 신규 2개), 그리고 무엇을 왜 고쳤는지 한 줄씩 정리한 목록.

### 1. 계약 반영

`docs/contracts/openapi3-server.yml` → `heymoa-web/openapi3.yml`, `docs/contracts/asyncapi-web-server.yml` → `heymoa-web/asyncapi.yml`.

**`/internal/**` 3경로는 제외한다.** `POST /internal/v1/callbacks/analyses/{analysisId}`, `GET /internal/v1/notes/{noteId}/context`, `GET /internal/v1/workspaces/{workspaceId}/notes` — 셋 다 heymoa-ai가 호출하는 경로다. 남겨두면 orval이 브라우저가 절대 부르지 않는 훅을 만들고 MSW 핸들러까지 붙는다. 제외한 사실과 이유를 `openapi3.yml` 헤더 주석에 남긴다. 결과는 34경로이고, 그중 **web이 새로 쓸 수 있게 되는 것이 21개**다.

`asyncapi.yml`은 전사 STOMP만 갖고 있는데 새 파일은 채팅 SSE까지 합본이다. 통째로 교체한다.

### 2. orval 재생성

`pnpm orval`. 기존 `orval.config.ts`를 그대로 쓴다 — msw·faker 생성기가 이미 켜져 있다.

생성물이 늘어나는 태그: AgentChats, NoteSharedChat, Notifications, WorkspaceInvitations, WorkspaceMembers, WorkspaceIntegrations, Analyses, Meetings.

**SSE 경로 두 개는 orval이 쓸모 있는 훅을 만들지 못한다.** `POST /v1/agent-chats/{chatId}/messages`와 `POST /v1/notes/{noteId}/chat/messages`는 응답이 `text/event-stream`이라 생성된 훅은 스트림을 못 읽는다. 이 둘은 기존 `lib/api/sse.ts` 유틸(APP-118 산출물) 위에 손으로 클라이언트를 만들고, 생성된 훅은 쓰지 않는다. 그 사실을 `lib/api/` 안에 주석으로 남겨 다음 사람이 생성 훅을 찾다 헤매지 않게 한다.

### 3. MSW 목

Vercel `dev` 배포가 이 목으로 돈다. 목의 충실도가 곧 데모의 충실도다.

**`lib/mocks/db.ts` 상태 확장** — 초대, 알림, 멤버, 연동, 개인 채팅 세션, 공유 채팅(잠금 포함), 승인, 분석, 회의 상태. 전이가 실제로 돌아야 한다:

- 초대 생성 → 대기 목록에 뜸 → 수락 → 멤버 목록 +1, 초대가 목록에서 사라짐
- 알림 클릭 → `unreadCount` 감소
- 회의 종료 → 분석 PENDING → (폴링) → 분석 완료, 요약 3종 렌더
- 승인 → 도구 실행 결과 → 채팅 히스토리에 기록 남음
- 연동 연결 → 상태가 "연결됨"으로, 연결한 사람·시점 표시

**REST 핸들러** — `lib/mocks/rest-handlers.ts`에 명시적 override로 추가한다. faker 기본값은 `success: false`를 무작위로 만들어 인증을 깨므로 금지(AGENTS.md).

**SSE 스트림 핸들러(신규 파일)** — 채팅 응답을 토큰 단위로 지연 발행한다. `message_start` → `token`×N → (`tool_call_start` | `tool_approval_request` → 승인 대기 → `tool_approval_resolved`) → `tool_call_result` → `token`×N → `message_end`. 승인 대기는 승인 API 호출이 올 때까지 열어 두고 `: keepalive` comment를 흘린다.

**목이 반드시 표현해야 하는 실패** — 빠지면 데모가 행복 경로만 보여주고, 화면 구현 단계에서 처리 코드를 쓸 근거가 없다.

| 실패 | 경로 | 화면에서 보여야 하는 것 |
|---|---|---|
| 409 입력 잠금 | `POST /v1/notes/{noteId}/chat/messages` | "OO님이 입력 중" + 입력창 비활성 |
| 409 회의 비ACTIVE | 같음 | "중지 중에는 개인 챗봇을 이용하세요" |
| 403 관전자 승인 시도 | `POST /v1/agent-chats/{chatId}/approvals/{approvalId}` | 승인 권한 없음 안내 |
| 404 만료된 승인 | 같음 | 카드 무효화 + 사유 |
| **종료 이벤트 없이 SSE 끊김** | 두 스트림 경로 | 재시도 UX. 처리 안 하면 영원히 로딩이다 |
| OAuth 리다이렉트 | `/v1/workspaces/{id}/integrations/{provider}/authorize` | 아래 참조 |

마지막 SSE 실패는 `docs/contracts/asyncapi-web-server.yml`이 "스트림이 끝나는 세 번째 경로"로 명시한 것이다. 동시 스트림 상한 초과·upstream 유휴 60초·입력 잠금 상실 시 server가 종료 이벤트 없이 연결만 끊는다.

**OAuth는 MSW로 완전히 재현되지 않는다.** authorize는 외부 도메인으로 리다이렉트하고 callback이 돌아오는 흐름이라 서비스 워커가 가로챌 수 없다. 목 환경에서는 authorize 클릭 시 **목 전용 승인 화면**을 우리 도메인에 띄우고 거기서 callback 경로로 돌려보낸다. 이 화면도 5단계 디자인 대상이다.

**Playwright는 이 단계에서 도입한다.** 목이 막 생긴 지금이 브라우저 서비스 워커 경로를 처음 검증할 수 있는 시점이고, 이때 넣어야 4·5단계가 같은 검증 명령을 쓴다. 상세는 아래 성공 기준의 "Playwright 스모크" 절.

### 4. API 맥락 맵

`heymoa-web/docs/api-surface.md` 신규. 6개 디자인 에이전트의 공통 입력이며, 화면 구현 단계에서도 계속 쓰인다.

여섯 에이전트가 같은 문서를 읽으므로 형식을 고정한다. **경로당 한 행의 표**:

| 경로 | 훅 | 쓰는 화면 | 응답이 만드는 상태 | v4 프레임 ID |
|---|---|---|---|---|

상태 칸에는 로딩·빈·에러·권한없음 중 그 경로가 실제로 만드는 것만 적는다. 프레임이 없으면 `없음`으로 적는다 — **`없음`의 집합이 곧 5단계의 작업 목록이다.**

여기에 **shadcn 프리미티브 매핑표**를 함께 둔다. 승인 카드는 Card + Button, 알림 벨은 DropdownMenu, 설정 탭은 Tabs, 잠금 안내는 Alert 같은 식이다. 디자인 에이전트가 각자 다른 프리미티브를 고르면 구현 단계에서 여섯 갈래로 갈라진다.

### 5. v4 디자인

**`mvp.pen`은 git으로 관리되지 않는다.** 이 사실이 두 가지를 강제한다.

- **작업 전 스냅샷을 뜬다.** `mvp.pen`을 `mvp.pen.bak-2026-07-23`으로 복사한 뒤 시작한다. 에이전트 6개가 동시에 쓰는 파일이라 한 번 망가지면 되돌릴 방법이 없다. 행 하나가 끝날 때마다 백업을 갱신한다.
- **결정은 git 안에 적는다.** `.pen`에만 남은 디자인 결정은 web 구현 단계에서 근거를 잃는다. 각 에이전트는 자기 행의 변경을 `heymoa-web/docs/design-decisions.md`(신규, git 관리)에 기록한다 — 무엇을 추가했고, 어떤 프레임 ID이며, 어떤 API·상태에 대응하고, 어떤 shadcn 프리미티브로 구현될 것인가. 화면 구현 이슈(APP-111~117)가 읽을 문서는 `.pen`이 아니라 이것이다.

**판정 기준은 계약 정합성이다.** 34경로가 만드는 화면과 상태가 v4 프레임에 빠짐없이 있고, 어긋난 것이 없으면 끝이다. 시각 언어는 v4의 것을 잇되, 4단계에서 근거가 나오면 재구성해도 된다.

에이전트 6개가 행 단위로 병렬 작업한다. `.pen` 동시 편집 충돌을 행 분할로 막는다 — 각 에이전트는 자기 행의 프레임만 만진다.

| 에이전트 | 행 (y좌표) | 프레임 수 |
|---|---|---|
| 1 | note-hub (20000) + side panel (32000) | 8 |
| 2 | 회의 종료·요약 (22000) | 4 |
| 3 | 공유 챗봇 (24000) | 3 |
| 4 | 개인 챗봇 (26000) | 3 |
| 5 | 도구 승인 (28000) | 3 |
| 6 | 연동·알림·멤버 (30000) | 4 |

각 에이전트가 받는 것: 자기 행의 현재 프레임 ID 목록, 4단계 API 맥락 맵에서 자기 행에 해당하는 부분, shadcn 매핑표, v4 그리드 규약(1440×900, x 간격 1840).

**실행 규약** — 여섯이 같은 파일을 쓰므로 규약이 흔들리면 결과가 여섯 갈래가 된다.

- **좌표**: 새 프레임은 `x = 그 행 마지막 프레임의 x + 1840`, `y`는 행 고정, 크기 1440×900.
- **재구성 경계**: 재구성은 **자기 행 안에서, 기존 v4 토큰·컴포넌트 범위 내로** 한다. 행을 넘는 변경(공용 `reusable` 노드, IA 자체)은 실행하지 말고 보고만 한다.
- **ground truth**: 팀 규칙은 "MSW로 실제 앱을 띄워 실측"이지만 5단계 화면 대부분은 아직 구현되지 않았다. **구현된 화면(노트·워크스페이스·설정)은 실측하고, 신규 화면은 계약 + 그 실측값에서 파생**한다. 파생 근거를 `design-decisions.md`에 남긴다. 추측으로 치수를 만들지 않는다.
- **병렬도**: 여섯을 한 번에 띄운다. 시작 직전 백업 스냅샷을 뜨고, 파일 손상이 확인되면 **복구 후 2명씩 3배치로 내려간다**. pencil의 동시 쓰기 안전성은 확인된 바 없고 git이 없어 백업이 유일한 복구 경로다.

## 브랜치

0~5단계를 **한 브랜치**에서 진행한다. 0단계에서 이슈가 재배치되면 확정된 번호로 브랜치를 다시 명명한다. `heymoa-web`은 PR 없이 `dev`로 squash-merge한다(AGENTS.md).

## 성공 기준

명령 하나로 도는 것부터. `test:e2e`는 3단계에서 생기므로 0~2단계에서는 그 앞까지만 돈다.

```bash
pnpm orval && pnpm test:run && pnpm lint && pnpm build && pnpm test:e2e
```

단계별 판정:

**0. 재배치** — APP-110 본문에 삭제된 파일 참조가 0건이다. 신규 이슈 2개(4·5단계)가 APP-102 하위에 생성됐다. APP-111~117 일곱 개가 5단계 이슈에 블록됨으로 표시됐다. 무엇을 왜 고쳤는지의 목록이 남았다.

**1. 계약** — `openapi3.yml`이 34경로이고 `/internal`이 0건이다. `asyncapi.yml`에 채팅 SSE 채널 2개가 있다.

**2. orval** — `pnpm orval`이 무오류로 끝나고 생성물에 `/internal` 훅이 0건이다. `lib/api/openapi-contract.test.ts`·`contract-consistency.test.ts`를 34경로로 확장해 이 둘을 테스트로 고정한다.

**3. 목** — 3단계 표의 전이 5개가 각각 테스트 하나씩 통과한다. SSE는 이벤트 순서, 승인 대기 후 재개, **종료 이벤트 없이 끊기는 경로** 세 가지가 통과한다. 실패 6종 목이 전부 존재한다. 그리고 **`pnpm test:e2e`가 존재하고 통과한다** — 이 단계에서 Playwright를 도입하므로, 여기부터 아래 검증 명령이 온전히 돈다.

**4. 맥락 맵** — 34경로가 전부 표에 있다. `v4 프레임 ID`가 `없음`인 행의 목록이 확정되고, 그것이 5단계의 입력이 된다.

**5. 디자인** — 맥락 맵의 `없음`이 0건이 된다. `design-decisions.md`에 여섯 행의 기록이 모두 있고, 각 신규·수정 프레임마다 **대응 API · 응답이 만드는 상태 · shadcn 프리미티브**가 적혀 있다. 공용 `reusable` 노드 변경은 0건이다. 행별로 `snapshot_layout`으로 구조를, 스크린샷으로 시각을 확인한다.

### Playwright 스모크 (3단계에서 도입)

vitest는 jsdom이라 **MSW의 브라우저 서비스 워커 경로(`worker.start()`)를 한 번도 지나지 않는다.** Vercel `dev` 배포가 정확히 그 경로로 도는데도 그렇다. 여기를 Playwright가 덮는다.

`@playwright/test` 설치, `playwright.config.ts`, `pnpm test:e2e` 스크립트, `e2e/` 디렉터리를 만들고 `NEXT_PUBLIC_API_MOCKING=enabled`로 dev 서버를 띄워 검증한다.

- 앱이 뜨고 콘솔 에러가 없다
- MSW 워커가 활성화된다
- 기존 화면(워크스페이스 목록·노트 상세·설정)이 목 데이터로 렌더된다
- **신규 훅 하나가 브라우저에서 목 응답을 받는다** — 예: 알림 목록. 3단계 목이 서비스 워커 경로에서도 도는지의 증거다

시각 회귀(스크린샷 비교)는 넣지 않는다. 화면 구현 goal마다 baseline을 갱신해야 해서 지금 넣으면 내내 시끄럽다. 이 스모크가 다음 goal들의 baseline이다.

### 리뷰 게이트 — 로컬 codex

각 단계의 코드가 커밋된 뒤, `dev` 대비 변경분을 로컬 codex로 리뷰한다.

```bash
codex exec review --base dev
```

**로컬 `codex exec review`만 게이트다.** PR 원격 Codex 리뷰는 요청하지도 반영하지도 않는다 —
이 저장소는 PR 없이 `dev`로 squash-merge하는 흐름이고, 게이트를 한 곳으로 모아야 판정이 흔들리지 않는다.

돌리는 시점은 셋이다.

1. **목 단계(3단계)를 끝낸 뒤** — 상태 전이와 SSE가 이 작업에서 가장 로직이 많은 곳이다
2. **Playwright 도입 뒤** — 새 도구·설정이 들어온 직후
3. **`dev`로 squash-merge하기 직전** — 브랜치 전체를 한 번

지적은 셋으로 가른다. **고친다 / 근거를 적고 넘어간다 / 별도 이슈로 뺀다.** 어느 쪽이든
`docs/codex-review-2026-07-23.md`에 한 줄씩 남긴다 — 넘어간 판단이 기록에 없으면 다음 사람이
같은 지적을 다시 받는다.

계약 미러(`openapi3.yml`·`asyncapi.yml`)와 orval 생성물(`lib/api/generated/**`)에 대한
지적은 **반영하지 않는다.** 손으로 고치면 다음 갱신에서 되돌아가고, 진짜 문제라면 고칠 곳은
heymoa-server다. 그런 지적은 기록만 하고 서버 이슈로 올린다.

### 요청 코드 사용 판정

"모든 API 호출은 orval 생성 훅으로"는 AGENTS.md 규칙인데 **강제 수단이 없다. 이번에도 문서로 둔다** — lint 규칙을 넣지 않으므로 새 위반은 리뷰에서만 잡힌다. 대신 규칙을 정확하게 만든다.

현재 AGENTS.md는 "직접 `fetch()` 금지"라고만 해서 `lib/api/sse.ts`가 규칙 위반처럼 읽힌다. SSE 두 경로는 생성 훅으로 처리할 수 없으므로 **예외를 명시**한다: 예외는 `lib/api/fetcher.ts`와 `lib/api/sse.ts` 둘뿐이고, 그 밖에서 API 경로로 직접 `fetch`하지 않는다.

## 위험

- **`.pen` 동시 편집** — 행 분할로 막지만, 에이전트가 공용 컴포넌트(사이드바·툴바 등 `reusable` 노드)를 고치면 충돌한다. 공용 컴포넌트 수정은 에이전트에게 금지하고, 필요하면 보고만 받아 순차로 반영한다. git이 없어 충돌을 사후에 감지할 수단도 없으므로, 백업 스냅샷이 유일한 복구 경로다.
- **생성물 규모** — 24경로가 늘면서 orval 산출물이 크게 늘어난다. `pnpm build` 시간과 번들 크기를 한 번 확인하고, 문제가 되면 태그별 코드 분할을 검토한다.
- **목과 실제 서버의 괴리** — 목이 통과해도 실제 서버에서 깨질 수 있다. 목은 계약(`openapi3.yml`)의 example과 스키마에서만 값을 만들고, 계약에 없는 필드를 임의로 넣지 않는다.
