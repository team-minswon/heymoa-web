# APP-156 챗봇 2종 재구성 + MSW 응답 풀 설계

**목표:** 공유 챗봇(노트 트레이)과 개인 챗봇(플로팅)이 같은 대화 UI 프리미티브를 쓰게 하고, MSW 챗 응답을 시드 기반 풀에서 결정적으로 뽑아 실제 밀도를 낸다.

**입력:** 프레임 `jsz26`(공유)·`lbdN9`(개인)·`I49sOL`/`d6dtU`(승인), 현재 `chat-thread`·`shared-chat-panel`·`personal-chat`·`lib/mocks/chat-stream.ts`.

**범위 밖:** 계약, SSE 경로(`postEventStream`), 알림·사이드바, 도구 승인/입력 잠금/ACTIVE 게이트 동작(유지).

## 자문자답으로 잡은 것

### 메시지 행은 이미 공유, 입력부만 갈라져 있다

실측: 두 챗봇 모두 `ChatThread`로 메시지 행을 그린다(공유 ✓). 그러나 **입력 form은 각자 구현**한다 — `shared-chat-panel`의 `Composer`(active 케이스)와 `personal-chat`의 인라인 form. 둘은 클래스·구조가 거의 동일하다(`border-t px-5/6 py-4` + `Input` + streaming ? 중지 : 보내기). 차이는 padding·placeholder·footer(개인의 승인 대기 안내)뿐. → **공유 프리미티브 `ChatComposer` 추출.** 두 곳이 이걸 쓴다. padding 단일화(px-5), 나머지는 props(placeholder·footer·onStop).

공유 챗봇의 phase 게이트(중지·미시작·관전 → `ComposerNotice`)는 공유 스코프 고유라 `shared-chat-panel`에 남고, active 케이스만 `ChatComposer`를 쓴다.

### 보내기 버튼이 프레임과 다르다

프레임(jsz26·lbdN9): 전송 버튼은 원형 다크 + ↑(ArrowUp). 코드는 사각 `size=icon` + `Send`(종이비행기). → `ChatComposer`에서 `rounded-full` + `ArrowUp`로 정합(중지도 원형). aria-label(보내기·중지) 불변 — 테스트 안 흔들림.

### MSW 챗 응답이 짧은 고정값이라 밀도를 못 본다

`chat-stream.ts`의 일반 응답이 `"회의에서 정한 내용을 정리했습니다."` 한 줄 고정이다. → **길이 있는 답변 풀**(2~4문장, 회의 Q&A 톤)에서 뽑는다. 순수 함수의 결정성을 지키려 입력(`chatId:turn:message`)에서 파생한 시드로 `faker.seed()` 후 `faker.helpers.arrayElement`(db.ts 패턴). 같은 입력 → 같은 답, 다른 턴 → 다른 답. `message_end.content`는 토큰 이어붙임과 동일(계약) — 기존 방식 유지. 승인/에러/끊김 경로는 불변.
